# Fortech — Transaction Analytics Deep Dive

## Domain (актуализированный)

- Реальный домен: Betting-сервис (ставки)
- Домен для резюме: Loyalty/Rewards Platform (платформа лояльности с баллами, транзакциями)
- Причина замены: предпочтение пользователя

## Scale

- ~1000 пользователей
- Высокая нагрузка, постоянные транзакции
- Реплики на дорогом железе (высоконагруженная инфраструктура)

## Case: Event-Driven Analytics (расширенный)

- Problem (S/T):
  - Аналитика для менеджеров/партнёров строилась через тяжёлые SQL JOIN-запросы к PostgreSQL
  - Latency до минут
  - Менеджеры не могли получить данные в реальном времени

- Users:
  - Менеджеры-аффилиаты (партнёры), которые приводили пользователей
  - Смотрели статистику по транзакциям пользователей

- Action (A):
  - Main Service → RabbitMQ → Consumer → ClickHouse
  - OLTP/OLAP разделение
  - Real-time event-driven pipeline

- Result (XYZ):
  - Latency: минуты → секунды
  - Менеджер заходит в админку и видит всё в real-time
  - Self-service аналитика без нагрузки на основную БД
  - Аналитики благодарили и хвастались удобством

## Key Quote
> "Он просто заходил, и у него уже есть всё в интерфейсе, прям заходит и смотрит всё в real-time"

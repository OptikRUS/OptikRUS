# Compel — Search UX Discovery: Full Story (Updated 2026-03-26)

## Case: От кэширования к пониманию пользователя

### Situation
- Поиск по наименованию электронных компонентов (RC073, и т.п.)
- Поиск начинался от 3 символов, normalized (lowercase, strip)
- При определённых запросах БД возвращала ~1.5M записей
- Приложение зависало, пользователь ждал минуту+

### Task (как пришла задача)
- "Сделать кэширование для тяжёлых компонентов"
- Задача пришла как техническое решение, не как бизнес-проблема

### Action — Первоначальное решение (кэш)
- Реализовал in-memory cache:
  - Прогрев при запуске приложения
  - Механизм eviction
  - Возможность добавления новых компонентов в кэш
- Проблема: на Stage/Dev были другие данные, чем на Prod (другой импорт, другие таблицы)
- Предварительно сделал рефакторинг, убрал лишние таблицы

### Action — Discovery момент
- На одной из встреч уточнял требования по поиску
- Узнал ключевой факт: **пользователи копируют и вставляют** названия компонентов
- Это типичное поведение на подобных B2B платформах
- Никто не набирает посимвольно — копируют из спецификаций, datasheet'ов

### Action — Изменение подхода
- "Продал" бизнесу новое решение:
  - Если пользователи copy-paste → минимум 3 символа не нужен
  - Никто не будет листать 1.5M результатов — это бессмысленно
  - Ограничили результаты до 1000
  - При слишком широком запросе → "Уточните запрос"
- Отказались от кэша

### Result
- Простое и дешёвое решение вместо сложного кэша
- Пользователи не заметили изменений (их UX паттерн не изменился)
- Время разработки сэкономлено: кэш с прогревом → простое ограничение
- Производительность решена без технического overengineering

### Key Insight
> "Understand the user before engineering the solution"
> Задача пришла как "сделай кэш", но реальное решение было в понимании UX паттерна.
> Copy-paste поведение полностью изменило технический подход.
